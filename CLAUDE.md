# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Commands

- `npm run build` ‚Äî Rollup bundles `src/main.ts` into `dist/main.js` + source map.
- `npm run watch` ‚Äî Rollup in watch mode.
- `npm run deploy` ‚Äî Bumps patch version, builds, uploads to Screeps via `scripts/deploy.mjs` (reads `SCREEPS_TOKEN` and `SCREEPS_BRANCH` from `.env`).
- `npm run localdeploy` ‚Äî Bumps patch version and builds only (no upload).
- `npm run lint` ‚Äî ESLint over `src/`.
- `npm run format` / `npm run format:check` ‚Äî Prettier.
- `npx tsc --noEmit` ‚Äî Type-check only.
- `npm test` ‚Äî Run all Vitest tests (`test/**/*.test.ts`).
- `npm run test:watch` ‚Äî Vitest in watch mode.
- `npm run test:coverage` ‚Äî Run tests with V8 coverage report.

Pre-commit hooks (husky + lint-staged) run `prettier --check` and `eslint` on staged `.ts` files automatically.

## Architecture

The AI is a single `loop` function exported from `src/main.ts`, called once per tick by the Screeps runtime. The loop is a linear pipeline of managers, each wrapped in `profile(name, fn)` for per-manager CPU accounting.

### Tick pipeline (order matters)

1. `initMemory()` ‚Äî one-shot Memory shape init per global reset.
2. `resetTickCache()` / `resetTraffic()` / `resetIdle()` ‚Äî clears per-tick caches.
3. `runDefense` ‚Äî scans hostiles, updates threat fields, activates safe mode on perimeter breach. Records hostile intel and logs combat events to `Memory.combatLog`. Runs first so spawner and towers see the same threat view.
4. `runSpawner` ‚Äî refreshes room plans, selects remote rooms, rebuilds the spawn queue. Bootstrap economy (harvesters) until first source container; then miner economy (miner + hauler + upgrader). Key invariants:
   - Hauler count is distance-scaled per source, **capped at `MAX_HAULERS_PER_SOURCE`** ‚Äî prevents a distant/swampy source saturating spawning.
   - Builder count is 0 when no construction sites exist (not 1 ‚Äî that path silently drained energy on fully-built rooms).
   - Defenders prepended dynamically; hunters queued after defenders at Priority 1.
   - Creep counts tracked per `homeRoom` so multi-room colonies don't share quotas.
5. `runLinks` ‚Äî source links ‚Üí storage link (primary) / controller link (secondary). Runs before rooms so creeps see fresh link state.
6. `runRooms` ‚Äî purges dead-creep memory, dispatches each creep via the `roles` registry. Roles register movement intents via `moveTo()` during this phase.
   6b. `resolveTraffic` ‚Äî resolves tile conflicts by priority and issues `creep.move()` calls.
7. `runTowers` ‚Äî focus-fires `pickPriorityTarget(room)` (threat-scored, not closest). Falls back to heal ‚Üí repair. Wall repair uses a per-RCL floor+cap formula (`wallRepairMax` in `towers.ts`). Ramparts on wall tiles are skipped (let them decay ‚Äî the wall is the barrier).
   7b. `runLabs` ‚Äî reaction selection (every 500 ticks), runs reactions when inputs loaded. `labFlushing` flag triggers hauler to clear stale minerals before loading new inputs. Boost lab (`boostLabId`) is skipped by the reaction loop. **Goal rotation**: `selectReaction` walks `REACTION_GOALS` and skips any goal whose stock ‚â• its `GOAL_CAPS` entry (hysteresis: resumes below cap√ó0.5), so production rotates GH2O ‚Üí defensive precursors (GHO2/KHO2/LHO2) once GH2O is satisfied instead of welding to GH2O forever. **Event-driven re-eval (v1.0.273):** in addition to the 500-tick periodic check, `runLabs` re-selects immediately when `isReactionViable()` (`reactions.ts`) says the active reaction can no longer run ‚Äî both inputs counted via `buildAvailableMap` (storage+terminal) **plus** whatever the two input labs already hold (so a loaded, mid-consumption batch still reads viable and doesn't thrash). Skipped while `mem.labFlushing` ‚Äî don't re-pick mid-flush, the established flush path already handles a changed selection. Without this, the hub could weld onto a stale pick (e.g. `activeReaction=ZK` with Z/K both exhausted) for up to 500 ticks. **Saturation-aware selection (v1.0.274):** `selectReaction` passes `INTERMEDIATE_SATURATION` to `findNextChainStep`, which **skips any step whose output is already ‚â• saturation** ‚Äî so the selector stops *making more* of an already-piled-up intermediate and advances the chain toward the goal. Fixes a live stall where the selector welded to `L+U‚ÜíUL` while UL sat stranded at ~14.7k, starving GH2O of the OH/GH production it needed (GH2O frozen at 264). Now UL (saturated) is skipped ‚Üí OH is made ‚Üí `GH2O` (GH+OH) becomes the highest viable step ‚Üí GH2O climbs to its `GOAL_CAPS` (4000). The goal compound itself is never wrongly skipped: its cap (4000) is below `INTERMEDIATE_SATURATION` (5000), and `isGoalSatisfied` governs it. The `saturation` arg is optional ‚Äî `getChainBuyNeeds`'s `findNextChainStep` call omits it (unchanged buying semantics).
   7c. `runTerminal` ‚Äî gated by `shouldRun({priority: THROTTLE_NORMAL})` (bucket ‚â• 5000; v1.0.260 ‚Äî was `THROTTLE_LOW`/8000, which silently disabled **all** terminal ops ‚Äî selling, mineral consolidation, colony energy sends, lab buying ‚Äî for 1900+ ticks live once the bucket settled at ~5700-6500, below the 8000 floor. Terminal logic is cheap, no pathfinding, just `Store`/market reads ‚Äî the lost economy throughput from skipping it outweighs the CPU saved, so it now runs whenever bucket ‚â• 5000 like labs/factory). Then gated purely on `terminal.cooldown === 0`, no `Game.time % INTERVAL` checks (v1.0.259: removed ‚Äî a deal's cooldown, often >10 ticks for distant buyers, routinely outlasted the 10-tick gap to the next aligned tick, so sell/ship/buy went silent for 900+ ticks live while manual deals worked instantly; cooldown alone is the natural rate limiter). Sells minerals above `MINERAL_TERMINAL_SELL_FLOOR`. **Sell floor must stay below `MINERAL_TERMINAL_CEILING`** ‚Äî equal values mean the ~5k pinned in storage prevents the terminal portion ever crossing its own line (observed: 23k H stuck unsold). Also runs `sendEnergyToColonies`: ships energy from surplus home rooms to needy colonies, score-sorted with hysteresis to prevent repeat sends on the same route. **Full-feeder lab model** (`isLabHub`/`getLabHubName` in `labs.ts` ‚Äî the owned room with the most labs, auto-detected): only the **hub** buys lab inputs (`buyForLabs`) and sells surplus; **feeder** rooms ship their minerals to the hub via `sendMineralsToHub` (the colony‚Üíhub mirror of `sendEnergyToColonies`) instead of selling. Feeders drain fully because their mineral storage floor is 0 (`mineralStorageFloor(room)` in `hauler.ts` returns `MINERAL_STORAGE_FLOOR` only for the hub) so all minerals flow to the terminal for shipment. `sellSurplus` is gated to `hub || no-hub-exists` (fallback preserves pre-hub behavior). Consolidates native H+O+Z across rooms onto one cluster so the hub stops buying O/Z it could get from colonies. `sellSurplus` processes `RESOURCE_BATTERY` first (v1.0.258) ‚Äî only one deal fires per cooldown window, and a high-revenue mineral (e.g. bulk H) would otherwise permanently starve the lower-revenue battery of its slot. **Batteries sell down to floor 0** (`BATTERY_TERMINAL_SELL_FLOOR`, v1.0.268) ‚Äî they're a pure for-sale product, so `MIN_DEAL_SIZE` (100) is the only gate. The floor previously reused `FACTORY_BATTERY_CAP` (400, the factory's *production* cap), which deadlocked sales: ~450 in the terminal gave surplus 50 < `MIN_DEAL_SIZE`, and the factory won't make more once `colonyEnergy` dips below `FACTORY_ENERGY_FLOOR` ‚Äî so terminal batteries sat permanently unsold with healthy buy orders on the book (observed live W43N58: 450 pinned at 178cr/unit demand). **Credit-drain guards (v1.0.272):** `buyForLabs` holds a `LAB_BUY_CREDIT_RESERVE` (50k) ‚Äî it skips when credits ‚â§ reserve and only ever spends the headroom above it (`affordableAmount = floor((credits ‚àí reserve)/price)`), so an expensive input never drains the wallet to ~0. And **`sellSurplus` never sells chain intermediates** (`getChainIntermediates()` in `reactions.ts` = compounds produced *and* re-consumed across the `REACTION_GOALS` chains ‚Äî UL/OH/ZK/G/GH/GO and the tier-2 boosts GH2O/GHO2/KHO2/LHO2); only raw native minerals, batteries, and the X-tier finals stay sellable. **Why:** all three native deposits (Z/H/O) deplete simultaneously and regen over 50k ticks; the ghodium branch's L is the priciest base mineral (~106cr, 3000-unit batch), and the single input-lab pair stalls G production so UL piled to ~13k and `sellSurplus` dumped it above the 10k floor at ~61cr ‚Äî buy-L‚Üímake-UL‚Üísell-UL-at-a-loss drained credits 1.5M‚Üí1 live. Holding intermediates means the L we buy is consumed into product, not sold back; the reserve caps the bleed. Net: the chain pauses gracefully while deposits regen instead of bankrupting the empire. **Saturation-aware buying (v1.0.273):** `getChainBuyNeeds` (`labs.ts`) now filters out any leaf input via `backedUpLeaves()` (`reactions.ts`) whose chain product has reached `INTERMEDIATE_SATURATION` (5000, `thresholds.ts`) ‚Äî e.g. stop buying L once UL has piled to 5k, because the bottleneck is lab time / a different input, not L. `buyForLabs`'s active-reaction fallback (fires when `getChainBuyNeeds` returns nothing but a reaction is running) applies the same guard: it won't buy `mem.activeReaction`'s inputs if that reaction's output is already ‚â• `INTERMEDIATE_SATURATION`. **Economy observability (v1.0.273):** `marketStatus()` console command (mirrors `boostStatus()`) reports credits + trend (from the new `Memory.creditHistory` ring, ~20 samples √ó `HEALTH_SNAPSHOT_INTERVAL`), the lab hub's active reaction and its stock vs. `GOAL_CAPS`, any stranded intermediates ‚â• `INTERMEDIATE_SATURATION`, what's currently gating `buyForLabs` (paused / credit-reserve / energy-floor / nothing needed), and recent buys/sells from `Memory._health`. **Stranded-intermediate release valve (v1.0.274):** the v1.0.272 "never sell intermediates" rule is now conditional ‚Äî `sellSurplus` still **holds** a chain intermediate by default, but if its combined storage+terminal stock is grossly overstocked (‚â• `INTERMEDIATE_RELEASE_THRESHOLD` = 10k = 2√ó saturation) it sells the terminal portion down to an `INTERMEDIATE_SATURATION` (5k) buffer to free frozen capital. Built because the saturation/reserve guards left ~14.7k UL (~900k credits) stranded with credits flat at 5 and no raw surplus to sell ‚Äî credits couldn't recover past the 50k reserve so buying stayed gated forever. The release **can't** reignite the buy-L‚Üísell-UL sawtooth: saturation-aware *selection* won't re-make the intermediate and `backedUpLeaves` won't let `buyForLabs` rebuy its leaf inputs while it's above saturation, so only capital the chain has already stopped consuming is released. **Decided against:** an `EconomyMission` / `TraderJoe`-style dedicated trading role, or a single mineral-budget scalar feeding all three managers ‚Äî the existing per-manager floors/ceilings (`MINERAL_TERMINAL_SELL_FLOOR`, `LAB_BUY_CREDIT_RESERVE`, `INTERMEDIATE_SATURATION`, `GOAL_CAPS`) already compose correctly and a unifying abstraction would cross into the shelved `LogisticsNetwork` mistake (engineering lesson #7) for marginal benefit ‚Äî `marketStatus()` gives the missing visibility without restructuring control flow.
   7d. `runFactory` ‚Äî RCL 7+ only. Compresses energy into batteries when storage > floor and battery stock < cap. **`FACTORY_ENERGY_FLOOR` (120k) must stay above the upgrader-expansion band** (`upgradersNeeded` mature ramp: 50k‚Üí1, 150k‚Üí2, 400k‚Üí3): when the floor sat at 50k it skimmed all surplus into batteries before storage could climb, capping storage below the 2nd-upgrader threshold and pinning W43N58 at 1 upgrader forever (observed live). The factory must only consume genuine surplus left after upgraders are funded.
8. `runConstruction` ‚Äî every 5 ticks. Placement priority: source containers ‚Üí controller container ‚Üí storage ‚Üí links ‚Üí extensions ‚Üí towers ‚Üí roads ‚Üí corridor roads ‚Üí remote roads ‚Üí terminal ‚Üí factory ‚Üí extractor ‚Üí mineral container ‚Üí labs ‚Üí ramparts ‚Üí perimeter walls ‚Üí perimeter ramparts. Terminal/factory/labs gated behind link completion (builders prioritise energy infra). `layoutPlan.version` invalidates cached layouts on bump. **Storage-access reservation (`layoutPlanner.ts`, v4):** before the extension stamp runs, up to `STORAGE_ACCESS_MIN` (3) walkable neighbours of the storage are added to `reserved` so extensions can never box in the storage ‚Äî the room's logistics hub needs multiple open access tiles or haulers/workers deadlock queueing for the single opening (observed live W44N57: storage strangled to 1 access tile, ~5 creeps frozen in the approach corridor). The `isAccessible` guard used for labs/factory doesn't cover this ‚Äî it only checks *cardinal* neighbours and accepts a single one, but storage range-1 reach includes diagonals. Already-built rooms aren't auto-un-boxed (the reservation only protects *open* tiles); fixing a boxed live room needs a manual extension `destroy()` + splice from the cached plan.
9. `runVisuals` ‚Äî opt-in `RoomVisual` overlay, gated by `Memory.visuals`.
10. `flushSegments` ‚Äî writes dirty `RawMemory.segments` entries.

Reordering has subtle effects: e.g. moving `runSpawner` ahead of `runDefense` lags defender production by a tick.

### Memory model (three layers)

- **`Memory`** ‚Äî hot, small. Creep roles, room threat fields, profiler stats, combat log, toggles. Keep it small ‚Äî Screeps JSON-parses it on every first access.
- **`RawMemory.segments` via `src/utils/segments.ts`** ‚Äî cold/large (room plans, scout reports). Lazy parse, dirty-flag writes. At most 10 active per tick; use `requestSegment(id)`. Segment 5 = neighbor intel.
- **`src/utils/tickCache.ts`** ‚Äî within-tick memoisation via `cached(key, fn)`. Cleared by `resetTickCache()`.

### Mission registry (`src/utils/missions.ts`)

Grouped goal-tracking lives in `Memory.missions`, a strictly-typed `MissionRegistry` (one sub-map per `MissionType`). Every record extends `MissionBase` (`type`, `id`, `status`, `createdAt`, `lastSynced`). Generic helpers ‚Äî `getMissionRegistry()`, `getMissionsOfType<T>(type)`, `garbageCollectMissions()` (registry-wide) ‚Äî sit under the type-specific API. **Invariant from the shelved hauler pool:** a mission governs spawn **quotas, lifecycle, and strategic gating** only ‚Äî it counts creeps (e.g. `RemoteMiningMission.haulerIds.length`) and never micro-dispatches a committed creep's per-tick task. Types: `remoteMining` (spawner-consumed; miner ownership stays canonical in `RoomMemory.sources[].minerName`) `colony` (the claim/expansion lifecycle `claiming‚Üíbootstrapping‚Üíactive`, owned by `colonyPlanner.ts`; legacy `Memory.colonies` folded in at Step 2 via `migrateColoniesToMissions()`), and `defense` (Step 3, formalize-only) ‚Äî one record per owned room's combat engagement, owned by `runDefense` (`defense.ts`): `active` on threat, `retiring` when it clears so the generic GC reclaims it. The DefenseMission mirrors the `combatActive` flag (retained for safe-mode/tower-drain logging ‚Äî a future cleanup could merge them); its `composition` snapshot is stamped by the spawner (disjoint field, avoids a defense‚Üîspawner import cycle). The fourth type, `transport` (`TransportMission`), is an **operator-created** manual cross-room energy/mineral delivery: couriers spawned from `destRoom` shuttle a resource out of `sourceRoom`'s storage/terminal into `destRoom`'s OWN storage. Created via the `deliverEnergy(source, dest, amount?)` console command; `targetAmount` is a **cap** ‚Äî `syncTransportMission` retires the mission on `deliveredAmount ‚â• target` **or** when the source is visible-and-empty with no courier carrying (source exhausted), so it never hangs. `amount` omitted ‚Üí `TRANSPORT_DRAIN_ALL` (a finite sentinel, **not `Infinity`** ‚Äî that JSON-serialises to `null` and corrupts Memory). The `courier` role does the per-tick haul (COLLECT from source `room.storage` ‚Äî owner-agnostic, so a reclaimed room's foreign storage is drained ‚Äî ‚Üí DELIVER into `myStorage(dest)`, crediting `deliveredAmount`). Spawner gates couriers from the dest room at a small distance-scaled cap, lowest priority. **Primary use:** drain a reclaimed room's previous-owner storage hoard into a mature colony ‚Äî the only way to extract it (no local sink, terminal is RCL6+) and it empties the husk so `cleanupClaimedRoom`/`placeStorage` build the source's own storage. `garbageCollectMissions` keeps a retiring transport while `courierIds` is non-empty. Adding a type = one field on `MissionRegistry` + a `getMissionRegistry()` sub-map guard. **Deferred:** cross-room defense dispatch and an `SKMission` type.

`src/utils/memoryInit.ts` guarantees `Memory.creeps` / `Memory.rooms` exist after reset.

### Adding a role

1. Create `src/roles/<name>.ts` exporting a `Role` (`run(creep): void`).
2. Define states as a `StateMachineDefinition` and call `runStateMachine(creep, states, defaultState)`.
3. Register in `src/roles/index.ts`.
4. Extend `CreepRoleName` union in `src/types.d.ts`.
5. Add a `SpawnRequest` in `buildSpawnQueue()`. Prefer a dynamic `*Needed(room)` function over a hardcoded count.
6. Set movement priority via `moveTo(creep, target, { priority: PRIORITY_* })`. Stationary roles call `registerStationary(creep, PRIORITY_STATIC)`.

The TypeScript union + `Record<CreepRoleName, Role>` registry means missing any step is a compile error.

### Shared utilities for roles

- **`gatherEnergy(creep)`** (`src/utils/sources.ts`) ‚Äî shared GATHER state: withdraws from logistics in miner economy, self-harvests in bootstrap. Returns `true` when full.
- **`deliverToSpawnOrExtension(creep)`** / **`deliverToControllerContainer(creep)`** (`src/utils/delivery.ts`) ‚Äî shared delivery with cross-tick caching. Multiple haulers spread across unclaimed targets to avoid convergence oscillation.
- **`ensureBoosted(creep)`** (`src/utils/boost.ts`) ‚Äî pre-role boost gate. Returns `false` while routing to a lab for `boostCreep`. **Fails open** (clears `boosts`, returns `true`) when no lab is found ‚Äî a creep must never stall permanently waiting for a boost. The reserved `boostLabId` is used only when its `boostCompound` matches the requested entry (else falls through to searching any stocked lab ‚Äî prevents a defender wanting KHO2 stalling at the GH2O lab). GH2O upgrader boost is complete and active; defenders (rangedDefender‚ÜíKHO2, healer‚ÜíLHO2) boost opportunistically off the rotating lab stock. **Total-attempt timeout (`BOOST_WAIT_TIMEOUT`=60):** a single per-boost-entry budget covering **travel to the lab AND any in-range wait** for the compound ‚Äî `boostWaitStart` is stamped at the top of the loop the first tick the entry is worked (not only on `ERR_NOT_ENOUGH_RESOURCES`), so it bounds **both** failure modes: (1) **in-range starvation** ‚Äî the compound sits in storage but no hauler ferries it in; (2) **travel deadlock** ‚Äî the creep can't physically reach the congested corner lab (observed live W43N58: an upgrader oscillated 40+ ticks rounding the 3√ó2 lab block from the far side; with no timer it parked near the labs forever = the "stuck near labs" symptom). After 60 ticks it fails open and works unboosted. A fresh upgrader spawns adjacent to the labs (spawn 16,31 ‚Üí boost lab 18,31) and boosts in ~3 ticks, so 60 is generous enough that normal operation never trips it. The hauler-side preempt (the link drain yields the boost lab when a creep awaits the compound, see below) is the primary fix for in-range starvation; this timeout is the backstop for both. Earlier live failure: 2 W43N58 upgraders idled ~500 ticks at the lab while 1.6k GH2O sat in storage because every hauler was perpetually committed to the storage-link faucet. **Reservation stability:** `upgraderBoostWanted`/`reserveBoostLab` (`spawner.ts`) count GH2O **already in the reserved lab + in transit** (`compoundInTransit` in `boost.ts`, summing the compound carried by creeps in-room) with **hysteresis** ‚Äî `BOOST_LAB_MINERAL_TARGET` (1500) to *start* a reservation, `BOOST_LAB_MINERAL_MAINTAIN` (500) to *hold* one. Without in-transit + lab counting, the instant a hauler withdrew GH2O to fill the lab, storage dipped below 1500, the lab unreserved mid-fill, `ensureBoosted` then found no lab and failed open ‚Äî so the lab could never be filled (the original recurring "labs fail to boost" root cause). Without hysteresis, a single boost consuming ~450 GH2O (30√ó15 WORK parts) dropped the total below 1500 and unreserved after every boost. **`compoundInTransit` is per-room+compound `cached()`** (tickCache) ‚Äî it scans all `Game.creeps` and is called once per owned room (`upgraderBoostWanted`) + up to 2√ó/hauler (`pickupBoostLab`); caching collapses ~44 scans/tick to one. **Observability (`boostStatus()` console command, always-on):** `recordBoostOutcome` bumps per-room counters in `Memory.boostStats` (`{failTimeout, failNoLab, failNoSupply, success, lastFailTick}`) at every `ensureBoosted` outcome. Because every failure mode now *fails open silently* (the timeout makes a chronic failure graceful but invisible ‚Äî the original W43N58 stall was found only by manual log-tailing), this is the production signal: a room where failures dominate `success` is flagged `‚öÝ FAILING`, surfacing chronic boost starvation without needing `Memory.boostDebug` verbose logs. **Decided against (do not re-chase ‚Äî engineering lessons #6/#7/#8):** (1) a **`BoostMission`** ‚Äî reservation is already two minimal RoomMemory flags and dispatch is `ensureBoosted`'s job; a mission would be either pure relocation (DefenseMission "formalize-only", marginal) or would cross into per-tick micro-dispatch (the shelved `haulerPool` mistake); (2) a structural **`haulersNeeded += 1` when `upgraderBoostWanted`** ‚Äî the demand-gated preempt already resolves the link-drain contention and an extra creep costs a dispatch slot against the 20-CPU cap (parked in `todo.md` as a tunable to revisit only if `boostStatus()` shows chronic starvation); (3) an **Overmind-style `LogisticsNetwork`** to replace the priority chain ‚Äî already tried (`haulerPool`) and shelved (worse convergence vs. emergent task-commitment), and a predictive optimizer is infeasible under 20 CPU.
- **`getColonyScore(room)`** (`src/utils/colonyPlanner.ts`) ‚Äî heap-cached score: `rclFactor √ó incomeRate √ó storageFactor`. Higher = worth investing in first. Used by upgrader count gates, colony energy sending, and visuals. Reads own storage via `myStorage(room)` so a reclaimed room's foreign hoard (`my:false`) does not inflate `storageFactor` and mis-rank a bootstrapping husk as the empire's richest room.
- **Hauler pickup priority** (`src/roles/hauler.ts`): committed target ‚Üí urgent responder ‚Üí **boost-lab preempt** ‚Üí large dropped energy ‚Üí lab flush/input/output ‚Üí **feeder-lab drain** ‚Üí storage link ‚Üí **terminal‚Üístorage restock** ‚Üí boost-lab service ‚Üí dropped energy/minerals ‚Üí ruins/tombstones ‚Üí full source containers ‚Üí **foreign-store drain** ‚Üí mineral container ‚Üí partial source containers ‚Üí factory batteries ‚Üí terminal minerals. The **terminal‚Üístorage restock** (`pickupTerminalEnergyToStorage`, `Memory.holisticEconomy` only) pulls energy from terminal into storage when storage drops below `upgradeBuffer(room)` (the RCL-keyed reserve: 25k at RCL6, 50k at RCL7) and the terminal holds more than `TERMINAL_ENERGY_FLOOR + TERMINAL_RESTOCK_MIN_BATCH` (15k+2k=17k). Terminal energy counts toward `colonyEnergy` so economy gates work correctly, but spawning and body-sizing read storage ‚Äî this restock makes storage track the actual budget. Never drains terminal below its 15k floor. Ranked after the link drain (primary pipeline) and before boost-lab service (fires only when healthy). The **boost-lab preempt** (`anyCreepAwaitingBoost` + `pickupBoostLab`, runs *before* committed/link work): when the reserved boost lab needs filling AND some creep in the room is parked waiting on that compound (`creep.memory.boosts` entry matches `boostCompound`), the hauler services the lab ahead of the storage-link drain. Without it the source-link faucet refills the storage link above the 200 drain threshold every tick, so with limited haulers all stay perpetually committed to the link drain (ranked above the normal boost-lab service) and the boost lab never gets filled ‚Äî upgraders idled ~500 ticks at the lab with GH2O sitting in storage (observed live in W43N58). The preempt is **demand-gated** (only when a creep actually awaits the compound) so it doesn't disturb the link pipeline during normal operation; `ensureBoosted`'s 60-tick total-attempt `BOOST_WAIT_TIMEOUT` is the backstop if a hauler still can't reach the lab. The **feeder-lab drain** (`pickupFeederLabs`) empties stale minerals left in a *feeder* room's labs (input labs included) so they flow storage‚Üíterminal‚Üí`sendMineralsToHub`: under the full-feeder model `runLabs` skips feeder labs (no reactions, no `labFlushing`) and `pickupLabOutput` skips input labs, so a room that ran reactions *before* becoming a feeder had its lab contents stranded with no evacuation path (observed live: W44N57 input labs pinned with Z+H). Gated on a hub existing elsewhere + this room not being the hub. **Runs in the lab-work block (gated on no `activeReaction`), not at the bottom of the chain (v1.0.268):** the old low rank meant a permanently-busy feeder's haulers never fell through to it, so the stale mineral sat indefinitely (observed live: W44N57 lab pinned at Z:1302 for hours despite the drain existing). It's a one-time finite job (~2 trips per lab), so the elevated priority self-limits ‚Äî it returns false once the labs are empty and never perpetually diverts haulers. **Critically, `runLabs` clears `activeReaction`/`labFlushing` on feeder rooms** ‚Äî without it `deliverToLabInput` (gated on `activeReaction`) would carry the just-drained mineral straight back into the input lab (infinite lab‚Üîhauler loop). The **foreign-store drain** (`pickupForeignStore`, reclaimed room's previous-owner storage via `lootTargetId`) ranks **low on purpose**: a foreign hoard is a *non-decaying reserve*, so it's drained only after decay-sensitive and fresh-income pickups (floor drops, full source containers). Placing it high starved the local economy ‚Äî source containers overflowed to cap and miner output decayed on the floor while haulers drained a bank that loses nothing by waiting (observed live in W42N59). See `hauler.ts` for the full chain and delivery logic (minerals to boost lab / lab input / storage / terminal; drop as last resort on young colonies). **SHELVED: `Memory.haulerPool`** must stay off ‚Äî pre-assignment via `src/managers/haulerPool.ts` conflicts with task-commitment (committed haulers ignore the assignment) and gave worse convergence in live testing. See `haulerPool.ts` header and `todo.md` Phase 6.

### Body scaling

`buildBody(pattern, energy, maxRepeats?)` in `src/utils/body.ts` scales with `energyCapacityAvailable` ‚Äî do not hardcode bodies. Specialised builders: `buildMinerBody`, `buildUpgraderBody`, `buildRemoteMinerBody`. Non-production roles capped via `maxRepeats` to control spawning cost.

### Economy model (`src/utils/economy.ts`, behind `Memory.holisticEconomy`)

A holistic energy model that replaces the scattered hardcoded storage floors with a single **`colonyEnergy(room)` = own storage + own terminal** budget (owner-agnostic via `myStorage`/`myTerminal`), spent top-down by priority. **Dark-deploy flag `Memory.holisticEconomy` (default off)**: every consumer branches `if (Memory.holisticEconomy) {budget path} else {original literal path}` ‚Äî flag-off preserves exact pre-refactor behaviour, so it ships safe and reverts from the console with no rollback. Built to fix three live failures: (1) the **collision** where the RCL6 mineral-mining floor (50k) equalled the 2nd-upgrader threshold (50k), so upgraders drained storage back below the line before a miner could commit; (2) **unconditional wall drain** ‚Äî towers held walls at 300k/1M HP regardless of how starved the room was; (3) **step-function upgrader cliffs**. Core API (all memoised per-room-per-tick via `energyBudget(room)`, `cached()`):
- **`economyStage`**: `bootstrap` (no miner economy) ‚Üí `growth` (RCL<6) ‚Üí `mature` (RCL6+) ‚Üí `saturated` (`colonyEnergy ‚â• 500k`, doubles upgrade output). Replaces scattered `rcl<6`/`minerEconomy` checks.
- **`upgradePower` / `upgradersNeeded` (mature)**: continuous `1 + floor(surplus / ENERGY_PER_UPGRADE_WORK(5k))`, √ó2 when saturated, then `ceil(power / upgraderWorkParts)` clamped `[1,4]` ‚Äî no cliffs, monotonic. `surplus = max(0, colonyEnergy ‚àí UPGRADE_BUFFER[rcl])` where buffer = `{5:10k,6:25k,7:50k,8:100k}`. The body-cap ladder in `spawner.ts` is **unchanged**; `upgraderWorkParts` mirrors it so count and body stay consistent.
- **`allowMineralMining`**: `rcl‚â•6 && stage‚âÝbootstrap && colonyEnergy > buffer + MINERAL_RESERVE_MARGIN(15k)`. The 15k margin is **deliberately ‚âÝ any upgrader threshold** so the collision cannot recur structurally (test-enforced in `economy.test.ts`).
- **`wallHpTarget`** (moderate-middle): `clamp(WALL_HARD_FLOOR[rcl] + floor(surplus√ó0.5), floor, WALL_CAPS[rcl])` ‚Äî floors lowered ~half (RCL6 300k‚Üí150k, RCL7 1M‚Üí400k) so a lean room holds a cheap target and lets storage climb; rich rooms scale up from genuine surplus.
- Factory (`colonyEnergy > FACTORY_ENERGY_FLOOR`), energy export (`HOME_SURPLUS_FLOOR` on `colonyEnergy`), and builder gates also read the budget. **Invariant preserved**: `UPGRADE_BUFFER[8]=100k < FACTORY_ENERGY_FLOOR=120k`, so the factory still only compresses post-upgrade surplus.

### Defense

- **Threat scoring** (`src/utils/threat.ts`): HEAL > CLAIM > RANGED_ATTACK > ATTACK > WORK (dead parts ignored). `pickPriorityTarget` weights threat + range-adjusted tower effectiveness ‚Äî prevents wasting fire on border creeps while closer hostiles roam.
- **Focus-fire is deliberate policy** (`managers/towers.ts`) ‚Äî closest-target fire lets healers keep attackers alive indefinitely.
- **Safe mode** triggers only when a `threatScore > 0` hostile is within range 5 of a spawn/storage/controller ‚Äî scouts don't burn a charge.
- **`defendersNeeded`** is tower-aware: returns 0 if energised towers can solo the threat and the enemy can't out-heal tower DPS. Towers fire regardless; this removes redundant spawn cost only.
- **`defenderComposition`** (`src/managers/spawner.ts`) ‚Äî returns `{ melee, ranged, healer }` counts by threat band, capped at 4 total. `defenderBoostsWanted(room)` (RCL 7+ and an aggressive *player* hostile present ‚Äî NPCs excluded) attaches T2 boosts to ranged/healer requests; melee stays unboosted (no TOUGH parts in its body). Fail-open, so unstocked rooms spawn unboosted.
- **`src/utils/neighbors.ts`** ‚Äî records hostile player intel in segment 5. Players classified `aggressive` cause remote planner to reject their observed rooms for 20k ticks.
- **Hunter role** (`src/roles/hunter.ts`) ‚Äî clears NPC Invaders from remote/transit rooms. Targets `'Invader'` owner only; never engages players. TRAVEL state waits for `isInRoomInterior` before transitioning to HUNT ‚Äî prevents work starting on a border tile the engine auto-evicts. Priority 1, one per infested room.
- **Remote threat flee** (`src/utils/remoteThreat.ts`) ‚Äî `handleRemoteThreat` runs first in miner/reserver/remoteBuilder/empty-remoteHauler roles: flee home and skip the state machine while the target remote room had a threat-scoring hostile within a cooldown. **Cooldown is owner-scoped**: NPC-only sightings (`'Invader'`/`'Source Keeper'`) use `NPC_HOSTILE_COOLDOWN` (50, comfortably above a hunter's ~20‚Äì30 tick kill) so a fresh miner doesn't time out waiting on an Invader the hunter already cleared; any player aggressor uses `HOSTILE_COOLDOWN` (300). The triggering sighting's classification is stored in `RoomMemory.hostileLastWasPlayer`; a missing flag (legacy memory) is treated as player (long cooldown) ‚Äî fail-safe. SK rooms exempt Source Keepers (keeperKiller handles them); haulers with cargo finish their delivery.
- **Perimeter defense** (`src/utils/perimeterPlanner.ts`) ‚Äî `computePerimeter` flood-fills inward from exits, stops at a Chebyshev `CORE_RADIUS` box around the spawn anchor; perimeter = exterior tiles bordering the interior. Walls on non-gate tiles, ramparts on gate tiles. Gate targets: sources outside core, distant controller, one per remote-room exit direction. `getPlannedReserved()` includes wall tiles so road pathfinding routes through gates. Stored in `RoomMemory.perimeterPlan`, invalidated by `PERIMETER_PLAN_VERSION` bump or remote-room change. `replanPerimeter(room)` forces a recompute; `construction.ts` consumes `perimeterPlan`. A terrain-aware min-cut variant (and a RoomVisual overlay) were prototyped and **removed** (v1.0.219‚Äì220) ‚Äî live validation showed the min-cut cost more wall than the radius ring for our room geometry; see git history (v1.0.216‚Äì218) before re-attempting against fresh data.
- **Combat log** (`src/utils/combatLog.ts`) ‚Äî ring buffer in `Memory.combatLog`. Console: `combatLog()`.

All roles use `src/utils/stateMachine.ts`. Each `StateHandler.run(creep)` returns a state name to transition or `undefined` to stay. State persisted in `creep.memory.state`. Falls back to default on rename (safe deploy). `onEnter()` called on transition.

### Movement & traffic

- **Always use `moveTo()`** from `src/utils/movement.ts` ‚Äî never `creep.moveTo()` or `creep.move()` directly.
- CostMatrix (`src/utils/trafficManager.ts`): roads=1, moving creeps=**0** (not soft-avoid ‚Äî cost-50 inflated corridors near idle hauler clusters, pushing PathFinder onto longer detours), stationary/hostile/impassable=255. Stuck-detection repath still uses cost-50 so recovery paths route around active blockers.
- **Obstacle-type construction sites = 255** (`applyConstructionSiteOverlay`). The Screeps engine refuses a move onto your *own* construction site of an obstacle structure (extension/link/tower/lab/terminal/storage/spawn/extractor/observer/powerSpawn/nuker/factory/constructedWall); road/container/rampart sites stay walkable. These sites aren't `FIND_STRUCTURES`, so they're absent from the base matrix ‚Äî applied fresh in the per-tick overlay (not the 20-tick base cache) so completed/cancelled sites update immediately. **Without this, every RCL upgrade froze creeps**: a batch of new sites lands on live corridors, PathFinder routes through them at terrain cost, the engine cancels the move (observed: a miner stuck beside a fresh link site in W44N57). The earlier version marked only extension sites at cost 10 ‚Äî passable-but-expensive ‚Äî which was wrong: the engine hard-blocks them, so 10 still routed creeps into a wall.
- **Active blocker pushing** (`pushBlocker`): nudges a lower-priority creep off the next tile. Best-effort, one push per tick per creep.
- **Stuck detection**: repath at 2 ticks stuck; force-repath with cache invalidation at 3 ticks, retry every 3.
- Room callback skips unseen rooms owned by other players ‚Äî avoids pathing into enemy tower range.
- Pass `range: 0` when the role must stand on a specific tile (default range is 1).
- **`isInRoomInterior(creep)`** ‚Äî true when ‚â•3 tiles from every border edge. Use in TRAVEL states: border-tile creeps can be auto-evicted by the engine.

### Remote mining

1. `scout` (1 MOVE) records source count, ownership, hostile presence, and source positions. Spawned only when below the storage-gated remote cap ‚Äî no permanent scout.
2. `selectRemoteRooms()` (`src/utils/remotePlanner.ts`) ‚Äî rejects owned/player-reserved/hostile rooms; scores candidates **distance-aware**: `sourceCount √ó SOURCE_SCORE_WEIGHT (100) ‚àí oneWayPathTiles`, so an extra source dominates (a 2-source room always beats a 1-source room) but among equal source counts the **closer** room wins, and any candidate beyond `REMOTE_MAX_PATH_TILES` (120 one-way) is hard-rejected. Path length comes from a cached PathFinder pass (`ensureRemotePathLength`, recomputed >5000 ticks stale; also feeds `remoteHaulersWanted`); when there's no spawn / the room is dark it falls back to a source-only score so selection never stalls. Selected up to a **myStorage-gated** cap (`remoteRoomCap`, hysteresis near 100k). **Gated on `myStorage(homeRoom)`, not `room.storage`** ‚Äî a reclaimed room's owner-agnostic foreign storage otherwise makes a fresh RCL2 colony spin up unaffordable remotes; with no own storage the selection is cleared to `[]`, freeing those remotes for **inter-colony de-confliction** (`claimedByCloserColony`): a remote is left to whichever storage-bearing sibling colony has the **shortest cached one-way path** to it (ties broken by room name), decided on distance alone ‚Äî **not** on who currently claims it, which was order-dependent and let two colonies grab the same room in one selection pass (live: W43N58 + W44N57 both took W44N58). So two colonies never double-mine one room. The scouted-hostile rejection is **NPC-aware**: a sighting flagged NPC-only (`scoutedHostileIsPlayer === false`) blocks selection for just `NPC_SCOUT_REJECT_TICKS` (300 ‚Äî hunters clear invaders fast), while a player sighting (or missing flag ‚Üí fail-safe player) blocks for `PLAYER_SCOUT_REJECT_TICKS` (1500). Known-*aggressive* players are independently hard-blocked 20k ticks by the `aggressiveInRoom`/`hostilesSeen` check. Mirrors the live-validated NPC-vs-player split in `remoteThreat.ts`.
3. `ensureRemoteRoomPlan()` ‚Äî scans visible remote sources; bootstraps from `scoutedSourceData` when dark.
4. Remote miners reuse the `miner` role with `targetRoom` set. They have 1 CARRY to build/repair their container. Pre-spawned `REMOTE_MINER_PRESPAWN_TICKS` before predecessor TTL to avoid coverage gaps.
5. `remoteHauler` picks up energy from the remote room, delivers home. Delivers `RESOURCE_ENERGY` **only** ‚Äî picking up non-energy minerals would trap them in the hauler permanently. **Not spawned until the remote source container exists** (`remoteMem.sources[].containerId`, set by the miner/`ensureRemoteRoomPlan`): before that the 1-CARRY miner is still building the container and produces little to haul, so pre-spawned haulers would just round-trip idle.
6. `reserver` (`[CLAIM√ó2, MOVE√ó2]`) ‚Äî 1 per remote room with a controller; net +1 reservation/tick doubles source capacity.
7. `remoteBuilder` ‚Äî builds/repairs remote roads. Temporary; stops spawning when roads are built and healthy.

`placeRemoteRoads` is **tunnel-aware**: its PathFinder pass stamps natural wall tiles at `TUNNEL_WALL_COST` (15√ó plain) via `applyTunnelWalls`, so a road routes through a wall only when it shortcuts a detour >~15 tiles longer. This overlay is local to road planning ‚Äî it never touches the creep-movement CostMatrix (creeps can't walk unbuilt walls). Built roads (cost 1) always beat tunnels, so completed paths are never re-dug.

`CreepMemory.homeRoom` = owner room. `CreepMemory.targetRoom` = operating room. Local miners have neither.

**Claiming** (`src/utils/colonyPlanner.ts`): `scoreClaimTarget` filters/scores a candidate (sources, distance, hostility, +5 for a mineral none of our rooms already mine); `findClaimCandidates()` ranks all scouted rooms, picking each one's nearest owned room as prospective home. Operator commits via `claim(roomName)`; inspect readiness with `claimCandidates()`. Lifecycle `claiming ‚Üí bootstrapping ‚Üí active` in `updateColonyStates`.

### Reclaiming a captured room

A room claimed from a previous owner is littered with their leftover structures. These are **owned by the old player and unusable by us**, and obstruct movement *and* our own construction. `cleanupClaimedRoom(room)` (`construction.ts`, called from `runConstruction` every 5 ticks for each owned room) handles this:

- **Destroys** foreign obstacle structures (`FOREIGN_OBSTACLE_TYPES`: spawn, extension, tower, link, lab, extractor, terminal, factory, observer, powerSpawn, nuker, storage) ‚Äî via `.destroy()`, which is free/instant and legal because we own the controller. **Critically, it destroys them even when they hold a little energy**: a foreign spawn/extension counts against *our* RCL structure-count limit, so a 300-energy leftover spawn returns `ERR_RCL_NOT_ENOUGH` on our own placement and hard-stalls the colony (observed live in W42N59 ‚Äî `placeColonySpawn` silently bailed every tick, builders dumped energy into the controller instead). Only `LOOTABLE_TYPES` (storage/terminal) **still holding resources** (`storeUsed > 0`) are spared for the drain path.
- **Keeps** roads & containers (ownership-neutral, reusable) and constructedWalls that are in the `perimeterPlan`; destroys unowned walls only when they sit on a planned layout tile.
- **Removes all foreign construction sites** (`FIND_HOSTILE_CONSTRUCTION_SITES` ‚Üí `.remove()`, legal in a room we own). We can't build another player's site and they block our own placement; an unfinished foreign site has no reuse value (unlike a built road).
- **`.destroy()` voids a structure's store, but `withdraw()` works directly on a foreign storage/terminal in a room we own** (live-validated W42N59, 2026-06-02: a normal hauler moved energy out, no WORK part needed). So a real hoard (e.g. W42N59's 607k-energy storage) is left standing (recorded in `RoomMemory.lootTargetId`) and **drained losslessly by existing haulers** via `pickupForeignStore` (`hauler.ts`) ‚Äî high in the pickup chain, alongside `pickupLargeDrop`. Energy drains at any RCL straight into the colony (spawn/extensions/controller container); a non-energy mineral is only pulled once we have an **own** storage/terminal to receive it (else it's left in place ‚Äî never trap a mineral in a hauler). Once the husk is **truly empty** (`storeUsed === 0`), the next cleanup sweep `.destroy()`s it, freeing the single storage slot ‚Üí `placeStorage` builds ours on the next tick. **Drain-to-empty, then swap ‚Äî lossless, no early-destroy lever, no dedicated creep.** (An earlier dismantle-based `looter` role was built on the *untested* assumption that `withdraw()` fails on a foreign store; that was disproven live and the role removed.)
- **Ownership guard:** `room.storage`/`room.terminal` are **owner-agnostic** ‚Äî in a reclaimed room they return the *previous owner's* structure. Use `myStorage(room)` / `myTerminal(room)` (`src/utils/ownership.ts`) at every ownership-sensitive site (deposits, placement gates, link anchoring, layout `storagePos`) so we never deposit our income back into the foreign store or skip placing ours. The drain path (`pickupForeignStore`) deliberately reads the foreign store directly via `lootTargetId`.

### Idle creep management

`src/utils/idle.ts` ‚Äî `markIdle(creep)` registers idle state, parks creeps away from the spawn cluster (deterministic per-name offset), and recycles chronically idle creeps. Builders/repairers/upgraders never idle. Haulers **do not recycle** ‚Äî the old recycle threshold churned haulers during normal idle gaps, costing more energy than it saved.

### Per-creep CPU throttle (`src/utils/creepThrottle.ts`, behind `Memory.creepThrottle`)

shard3 hard-caps every player at 20 CPU regardless of subscription/GCL ‚Äî the only path to more rooms is per-creep efficiency, not more CPU. Live (2026-06-13): 53 creeps / 4 rooms, bucket idles ~5300-6500 and drains ~0.38/tick on average; left unchecked it eventually hits 0, where the engine hard-throttles mid-loop (corrupt half-ticks). `shouldThrottleCreep(creep)`, called in `runCreeps` (`src/managers/room.ts`) right before role dispatch, probabilistically skips a creep's per-tick role logic once the bucket drops into a danger band ‚Äî a Hivemind-style Van der Corput even spread (`SPREAD`, ~256 entries) plus a per-creep heap offset (`creepOffset`, cleared on global reset) so skips decorrelate across creeps/ticks instead of all happening on the same tick. **This is a stability/floor-protector, not added capacity** ‚Äî it trades a little upgrade/repair/build throughput for a bucket that self-stabilizes above 0.

Role tiers (`Record<CreepRoleName, ThrottleTier | null>` ‚Äî compiler-enforced exhaustive):
- **NEVER** (`null`): `defender`, `rangedDefender`, `healer`, `hunter`, `keeperKiller` (combat acts every tick), `miner` (a skipped harvest is irreplaceable income), `claimer`, `colonyBuilder` (claim lifecycle is time-sensitive).
- **TIER_LIGHT** (`throttleAt: 2500, stopAt: 500`): `hauler`, `remoteHauler`, `courier`, `reserver`, `mineralMiner`, `harvester` ‚Äî income-adjacent logistics, only throttled when genuinely low.
- **TIER_HEAVY** (`throttleAt: 4000, stopAt: 1500`): `upgrader`, `repairer`, `builder`, `remoteBuilder`, `scout`, `dismantler` ‚Äî discretionary, shed first.

Both tiers' `throttleAt` sit **below** our ~5500 idle bucket on purpose ‚Äî at normal operation nothing throttles; the band only engages once a drain pushes the bucket below it. Decision order: flag off ‚Üí false; NEVER role ‚Üí false; border creep (`!isInRoomInterior`) ‚Üí false (engine eviction risk); emergency brake `Game.cpu.getUsed() > tickLimit*0.85` ‚Üí true; `bucket >= throttleAt` ‚Üí false; `bucket <= stopAt` ‚Üí true; otherwise probabilistic via `SPREAD`. Default off (dark-deploy); flip with `Memory.creepThrottle = true`. When on, logs `[throttle] skipped N/M creeps, bucket=B` every 100 ticks.

### Live inspection (context-cheap, no MCP)

`scripts/screeps-query.mjs` reads live bot state over the Screeps HTTP API, reusing the **same `.env` `SCREEPS_TOKEN`** as deploy (sent as `X-Token`; never printed, never a CLI arg). It exists to keep agent context small: the `screeps-mcp` read tools dump large raw payloads (console buffers full of log chatter, whole Memory subtrees) into context, whereas this prints only a scoped, already-filtered result.

- `node scripts/screeps-query.mjs mem <path>` ‚Äî GET a Memory subtree (dot path, e.g. `mem _health`, `mem boostStats`, `mem rooms.W43N58`), gzip-decoded. No game-tick latency, no size limit. **Preferred for anything that lives in Memory.**
- `node scripts/screeps-query.mjs probe <file.js>` ‚Äî run a single JS expression in-game (from a file in `scripts/probes/`), capture its return into `Memory._probe`, poll until fresh, print it. For live `Game.*` data not in Memory. **Constraints:** the Screeps console caps **expression** size (~1KB ‚Äî the script strips comments/indentation but not intra-line spacing, so keep probe files free of inline mid-code comments and double-spaces inside string literals); the **result** can be any size (stored in Memory, read back gzipped). The console command queue has variable latency (~12‚Äì60s).
- `node scripts/screeps-query.mjs run "<expr>"` ‚Äî same as `probe` but with an inline (quoted) expression. Because the result is unbounded, any console function works: `run "combatLog()"`, `run "colonies()"`, `run "status()"`, `run "boostStatus()"`, `run "marketStatus()"`. Also the way to do one-off writes/toggles, e.g. `run "Memory.profiling=true"`.

Two read-only telemetry rings let the script replace MCP for almost everything (see [[feedback_prefer_query_script_over_mcp]]):
- `Memory._health` ‚Äî compact health snapshot written every `HEALTH_SNAPSHOT_INTERVAL` ticks (`src/utils/healthSnapshot.ts`, called from the loop). Makes the **HealthCheck skill** a single `mem _health` read instead of three MCP console probes. Uses `myStorage`/`myTerminal` so reclaimed rooms report our structures.
- `Memory._errors` ‚Äî ring of the last 20 source-mapped errors caught by `ErrorMapper.wrapLoop` ‚Üí `recordError` (capped, each truncated to 1000 chars). `mem _errors` is the check for **errors thrown during normal operation**. **It does NOT cover post-deploy "did the new bundle load?"**: a bundle that fails to parse/load throws at module-load time *outside* `wrapLoop`, so it never reaches `recordError` or the ring ‚Äî that only surfaces in the live console stream. So **routine error check ‚Üí `mem _errors`; post-deploy load verification ‚Üí MCP `check_for_errors`** (the one legitimate remaining post-deploy MCP use, alongside tailing the live `console.log` stream of spawn/terminal chatter).

### Error mapping

`src/utils/ErrorMapper.ts` wraps the loop with a synchronous VLQ decoder (not `source-map` ‚Äî it's async/too slow) to map runtime errors to TypeScript source lines. Map cached across ticks, rebuilt on global reset.

## Engineering lessons

1. **External insights don't always translate** ‚Äî understand *why* something works elsewhere before porting; context beats the idea.
2. **Live validation is ground truth** ‚Äî unit tests pass and it still breaks; the Screeps environment surfaces edges no mock captures.
3. **Feature flags are worth the boilerplate** ‚Äî ship dormant, test safely, shelve without reverting; shelved code must explain *why* it's off and what it conflicts with.
4. **Respect existing architectural invariants** ‚Äî understand what a system already solves before replacing it.
5. **Measure before optimising** ‚Äî "looks bad visually" is not evidence; define a proxy metric first.
6. **Complexity has a carrying cost** ‚Äî a simpler slightly-suboptimal system usually beats a theoretically better complex one.
7. **Emergent solutions can be more robust** ‚Äî task-commitment outperformed a designed dispatch system; understand emergent behaviour before replacing it.
8. **Annotate shelved code thoroughly** ‚Äî the comment is the only thing preventing re-implementing the same mistake; include what, why, what it conflicts with, and where to look.
