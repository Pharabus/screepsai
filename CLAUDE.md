# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Commands

- `npm run build` — Rollup bundles `src/main.ts` into `dist/main.js` + source map. The Screeps runtime loads `dist/main.js` as the AI entry point.
- `npm run watch` — Rollup in watch mode.
- `npm run deploy` — Bumps patch version, builds, then uploads `dist/main.js` to Screeps world servers via `scripts/deploy.mjs` (reads `SCREEPS_TOKEN` and `SCREEPS_BRANCH` from `.env`).
- `npm run localdeploy` — Bumps patch version and builds only (no upload). Use when copying `dist/main.js` to a local Screeps server manually.
- `npm run lint` — ESLint over `src/`.
- `npm run format` / `npm run format:check` — Prettier.
- `npx tsc --noEmit` — Type-check only.
- `npm test` — Run all Vitest tests (`test/**/*.test.ts`).
- `npm run test:watch` — Vitest in watch mode.
- `npm run test:coverage` — Run tests with V8 coverage report.

Pre-commit hooks (husky + lint-staged) run `prettier --check` and `eslint` on staged `.ts` files automatically.

## Architecture

The AI is a single `loop` function exported from `src/main.ts`, called once per tick by the Screeps runtime. The loop is structured as a linear pipeline of managers, each wrapped in `profile(name, fn)` for per-manager CPU accounting.

### Tick pipeline (order matters)

1. `initMemory()` — one-shot Memory shape init (per global reset).
2. `resetTickCache()` — clears the transient per-tick memoisation map.
   2b. `resetTraffic()` — clears intent-based traffic manager state.
   2c. `resetIdle()` — clears the idle creep set for fresh per-tick tracking.
3. `runDefense` — scans hostiles, updates `RoomMemory.threatLastSeen` / `lastThreatScore`, activates safe mode on perimeter breach. Also calls `recordHostile()` for each hostile to update the neighbor-tracking segment, and `requestNeighborSegment()` so it's available next tick. Logs combat lifecycle events (`threat_appeared`, `threat_ended`, `safe_mode_activated`, `safe_mode_unavailable`) to the `Memory.combatLog` ring buffer (100 entries max) via `src/utils/combatLog.ts`. Runs first so the spawner and towers see the same threat view.
4. `runSpawner` — calls `ensureRoomPlan(room)` to refresh source/container/miner cache, runs `selectRemoteRooms` every 100 ticks, scans remote rooms via `ensureRemoteRoomPlan`, then **rebuilds the spawn queue per room** via `buildSpawnQueue(room)`. Uses bootstrap economy (harvester-based) until the first source container is detected, then switches to miner economy (miner + hauler + heavy-WORK upgrader). Hauler count: base 2 (linked sources add `min(linked,1)`, unlinked add a distance-scaled count per source — `ceil(roundTripTiles × rate ÷ haulerCarry)` — **capped at `MAX_HAULERS_PER_SOURCE` (5)** so a very distant source like W44N57's pathDist-100 swamp source can't demand 8+ haulers and saturate the spawn), +1 when mineral mining is active (mineral container present and mineral available), +1 per active remote room (remote energy arrives in bulk and needs dedicated distribution bandwidth). Upgrader count: split by colony maturity. **Hard floor** (all rooms): storage exists and stored < 5k → 1 upgrader below RCL 8 (body storage-capped to stay small), 0 at a built-out RCL 8 room. **Young colony (RCL < 6)**: three safety gates checked in order — (a) income gate: `getColonyScore(room) < YOUNG_COLONY_MIN_SCORE (20)` → return 1 (too income-starved to drain storage); (b) builder guard: construction sites exist and stored < 20k → return 1 (don't starve builders); (c) above both gates: 2 when stored < 15k, 3 when < 40k, 4 when ≥ 40k — aggressive ramp to push RCL fast. **Mature colony (RCL 6+)**: 1 when < 100k, 2 when < 200k, 3 when < 500k, 4 when ≥ 500k — deliberately conservative so storage builds surplus before adding upgrade drain. Upgrader bodies use a 3-tier energy cap to scale with reserves: young colony gets full body at 15k stored; mature colony at 50k stored; both cap at 600 below the first tier, 1100 below the second — small bodies during scarcity prevent a 15-WORK upgrader from draining storage faster than miners can refill it. Builder spawning is paused (0 requested) when all sources are linked and storage energy is below `STORAGE_ENERGY_FLOOR` (10k), and is also 0 when no construction sites exist (was previously 1 to idle-upgrade — that path silently drained energy on fully-built rooms). Harvester in miner economy: 0 when all sources have an assigned miner, 1 (emergency bootstrap) only when a source lacks a miner — the role retires once miner coverage is complete. Remote miners are pre-spawned 150 ticks before TTL expiry so source coverage doesn't gap while the old miner travels home to recycle. Remote mining roles (scout, remote miners, remote haulers) are appended at lowest priority in miner economy. Defender spawning is prepended dynamically when threats are active, with `defenderComposition(room)` selecting a mix of melee (`defender`), ranged (`rangedDefender`), and `healer` creeps based on threat band and hostile body composition. Hunter spawning (Priority 1, after defenders) queues one `hunter` per unique Invader-infested remote or transit room that lacks a hunter, using `buildHunterBody` scaled to `energyCapacityAvailable` — not spawned at all below 790 energy capacity. Creep counts are tracked per-room via `homeRoom` so multi-room colonies don't share quotas.
5. `runLinks` — transfers energy from source links to storage link (primary) or controller link (secondary). Runs before rooms so creeps see fresh link state.
6. `runRooms` — purges dead-creep memory, backfills `creep.memory.homeRoom` for any local creep lacking it (transition safety for creeps spawned before per-room tracking), then dispatches each creep to its role via the `roles` registry (`src/roles/index.ts`). Roles register movement intents via `moveTo()` during this phase. Per-creep calls are profiled as `role.<roleName>`.
   6b. `resolveTraffic` — processes all movement intents registered during `runRooms`, resolves tile conflicts by priority, and issues `creep.move()` calls.
7. `runTowers` — every tower in a room focus-fires `pickPriorityTarget(room)` (threat-scored, not closest). Falls back to heal → repair with a 50% combat energy reserve. Wall/rampart repair target uses `wallRepairMax(room)`: `Math.min(Math.max(WALL_FLOOR[rcl], stored × 0.5), WALL_CAPS[rcl])` — `WALL_FLOOR` is a per-RCL minimum (3=10k, 4=50k, 5=150k, 6=300k, 7=1M, 8=5M) that prevents paper-thin walls during lean periods; `WALL_CAPS` is the per-RCL maximum (3=10k … 8=50M). Ramparts co-located on wall tiles are skipped (let them decay naturally — the wall is the intended long-term barrier). Logs `tower_energy_low` to the combat log when any tower drops below 25% energy during combat.
   7b. `runLabs` — selects a reaction from available minerals (re-evaluated every 500 ticks), runs `outputLab.runReaction(inputLab1, inputLab2)` on all output labs each tick when input labs are loaded. Stores the active reaction in `RoomMemory.activeReaction`. When a reaction change leaves stale minerals in input labs, sets `RoomMemory.labFlushing` — hauler withdraws wrong minerals before loading new inputs, flag auto-clears when labs are clean. A lab reserved for boosting (`RoomMemory.boostLabId`) is skipped by the output-reaction loop so reactions never consume or overwrite its boost compound.
   7c. `runTerminal` — runs every 100 ticks. Sells terminal minerals above `MINERAL_TERMINAL_SELL_FLOOR` (10k) to the best buy order via `Game.market.deal()`. The sell floor is deliberately below `MINERAL_TERMINAL_CEILING` (20k, the combined storage+terminal hold/throttle cap used by the mineral miner) so a sellable band always exists — when both used the same 20k value, the ~5k pinned in storage meant the terminal portion could never cross its own sell line and minerals never sold (observed: 23k H stuck unsold at 2 credits). One sale per terminal per interval. Gated by minimum price (0.01) and energy cost check (`ENERGY_TERMINAL_BUFFER` = 5k reserved for transaction fees, `TERMINAL_ENERGY_FLOOR` = 15k below which hauler tops the terminal up from storage). Shared thresholds live in `src/utils/thresholds.ts`. Also runs `sendEnergyToColonies` every 100 ticks: ships `COLONY_SEND_AMOUNT` (10k) energy from a home room with ≥ `HOME_SURPLUS_FLOOR` (80k) storage to the highest-priority colony whose terminal is online but storage is below `COLONY_STORAGE_TARGET` (30k). Receiver selection is score-driven — candidates are sorted by `getColonyScore(room)` DESC, then storage ASC (most urgent first), then linear distance ASC (cheapest fee first). Hysteresis (`COLONY_SEND_HYSTERESIS_TICKS = 300`) prevents re-sending on the same home→colony route within 300 ticks of the last shipment; the first send on a new route is never blocked. Stays dormant for pre-RCL-6 colonies that have no terminal yet.
   7d. `runFactory` — runs at `THROTTLE_NORMAL`. RCL 7+ only. Level 0: produces `RESOURCE_BATTERY` (50 energy → 1 battery via `COMPRESS` action) when `room.storage` energy > `FACTORY_ENERGY_FLOOR` (50k) and battery stock < `FACTORY_BATTERY_CAP` (500). Sets/clears `RoomMemory.factoryRecipe` for hauler targeting. Hauler delivers energy to factory when recipe is active and picks up batteries for terminal/storage.
8. `runConstruction` — runs every 5 ticks. Places structures in priority order: source containers → controller container → storage → links → extensions → towers → roads → corridor roads → remote roads → terminal → factory (RCL 7+) → extractor → mineral container → labs → ramparts → perimeter walls → perimeter ramparts. Each `place*` function is RCL-gated. **Perimeter defense pipeline**: `placePerimeterWalls()` (RCL 6+) places walls on non-gate perimeter tiles; `placePerimeterRamparts()` (RCL 5+) places ramparts on gate tiles only at RCL 6+ (all perimeter tiles at RCL 5 when walls aren't yet available). Both are gated behind `PERIMETER_STORAGE_MIN` (20k) to prevent energy starvation. Gate tiles (passable by own creeps) never get walls. Non-gate perimeter tiles never get ramparts — walls are cheaper to maintain (no decay). `getPlannedReserved()` includes non-gate perimeter tiles so road pathfinding routes through gates automatically. Terminal, factory, extractor, mineral container, and labs are also gated behind link completion — they won't place construction sites while any link site is under construction, ensuring builders prioritize energy infrastructure. `layoutPlan.version` (tracked via `LAYOUT_PLAN_VERSION` in `layoutPlanner.ts`) invalidates cached layout plans on version bump, triggering an automatic one-time recompute per room. `placeCorridorRoads()` builds roads along the dx=0 and dy=0 corridors through the extension diamond for creep accessibility. `placeRemoteRoads()` uses cross-room `PathFinder.search` from the home spawn to each remote source, placing road sites incrementally (one per tick). Only targets reserved rooms with an active reserver. Extension slot limits per RCL (from `CONTROLLER_STRUCTURES`): 5/10/20/30/40/50/60 at RCL 2–8. At RCL 7 each extension holds 100 energy (up from 50 at RCL 6), at RCL 8 it's 200, so spawning capacity rises sharply even though slots grow by only 10 per tier. The layout planner pre-plans 70 extension positions (enough to absorb road-blocked slots and cover through RCL 8); `placeExtensions` only builds up to the RCL cap and falls through to a wider overflow search when plan positions are exhausted. **Auto-dismantle of blocking structures**: if a builder is in its BUILD state and finds `FIND_HOSTILE_STRUCTURES` of type `STRUCTURE_EXTENSION` or `STRUCTURE_TOWER` nearby (previous-owner structures from a room that was reclaimed), it dismantles them before building — this requires no energy and unblocks RCL-gated placement slots. The builder only targets `FIND_MY_CONSTRUCTION_SITES` for actual building, so it never accidentally builds on a previous owner's sites.
9. `runVisuals` — opt-in `RoomVisual` overlay, gated by `Memory.visuals`.
10. `flushSegments` — writes dirty `RawMemory.segments` entries and registers requested segments for next tick.

Reordering these has subtle effects: e.g. moving `runSpawner` ahead of `runDefense` would make defender production lag a tick behind sightings.

### Memory model (three layers, pick the right one)

- **`Memory`** — hot, small. Today: `Memory.creeps[name].role`, `Memory.rooms[name]` (threat fields, `perimeterPlan`), `Memory.stats` (profiler), `Memory.combatLog` (ring buffer, max 100 entries), and toggles (`profiling`, `visuals`). Screeps JSON-parses this blob on first access each tick, so keep it small. `perimeterPlan` (~800–1200 bytes per room) fits comfortably in main Memory without needing segments.
- **`RawMemory.segments` via `src/utils/segments.ts`** — cold/large data (room plans, scout reports, historical stats). Lazy parse on read, dirty-flag writes (only mutated segments serialise at `flushSegments()`). At most 10 segments may be active per tick; use `requestSegment(id)` to queue for next tick. Segment 5 is reserved for neighbor/hostile-player intel (`src/utils/neighbors.ts`).
- **`src/utils/tickCache.ts`** — within-tick memoisation, cleared by `resetTickCache()`. Use `cached(key, () => expensive())` when multiple managers need the same `room.find` / aggregate.

`src/utils/memoryInit.ts` guarantees `Memory.creeps` and `Memory.rooms` exist after a global reset so hot-path code doesn't need `??= {}` guards.

### Adding a role

1. Create `src/roles/<name>.ts` exporting a `Role` (`run(creep): void`).
2. Define states as a `StateMachineDefinition` (see `src/utils/stateMachine.ts`) and call `runStateMachine(creep, states, defaultState)` from `run()`.
3. Register it in `src/roles/index.ts`.
4. Extend the `CreepRoleName` union in `src/types.d.ts`.
5. Add a `SpawnRequest` entry in `buildSpawnQueue()`. Prefer a dynamic `*Needed(room)` function (see `buildersNeeded`, `repairersNeeded`, `upgradersNeeded` as examples) over a hardcoded `minCount`. Every role's count should reflect current room state so the economy self-balances.
6. Set appropriate movement priority via `moveTo(creep, target, { priority: PRIORITY_* })`. Stationary roles (miners) should call `registerStationary(creep, PRIORITY_STATIC)` in their harvest state.

The TypeScript union + `Record<CreepRoleName, Role>` in the registry means forgetting any of these steps is a compile error.

### Shared utilities for roles

- **`gatherEnergy(creep)`** (`src/utils/sources.ts`) — shared GATHER state logic used by builder and repairer. Withdraws from logistics in miner economy, self-harvests in bootstrap. Returns `true` when store is full. `withdrawFromLogistics` priority: storage (above `STORAGE_ENERGY_FLOOR` 10k) → storage link → dropped energy → any source container (>100 energy, including linked) → gives up (caller falls back to self-harvest).
- **`deliverToSpawnOrExtension(creep)`** / **`deliverToControllerContainer(creep)`** (`src/utils/delivery.ts`) — shared delivery helpers used by hauler and remoteHauler. Return `true` if a target was found. `deliverToSpawnOrExtension` opportunistically fills any adjacent (range 1) spawn/extension before pathing, then caches a movement target across ticks. Multiple haulers spread across unclaimed targets to avoid convergence oscillation.
- **`ensureBoosted(creep)`** (`src/utils/boost.ts`) — pre-role boost gate. Returns `true` when the creep has no pending boosts or all are applied (role proceeds); `false` while it still needs to reach a lab and `boostCreep`. Reads `CreepMemory.boosts` (a list of `{ part, compound }` set at spawn time) and resolves a lab via `RoomMemory.boostLabId` (reserved — waits even if understocked, since a hauler is filling it) or, failing that, any room lab already holding enough compound. Fails open (clears `boosts`, returns `true`) when no lab can be found or `boostCreep` returns an unexpected code, so a creep can never permanently stall waiting for a boost. **Boost framework Steps 1–3 complete and active in production.** Step 3: `upgraderBoostWanted(room)` in `spawner.ts` gates on RCL 7+, ≥2 output labs, GH2O stock ≥ 1500 across storage+terminal+the reserved boost lab, and storage energy > floor. (Counting the boost lab's own GH2O keeps the gate stable: reserving the lab moves up to 1500 GH2O out of storage into it, so a storage-only sum would drop to the threshold and flip-flop the gate the moment a boost is consumed.) `reserveBoostLab(room)` runs each tick in `runSpawner` to set/clear `boostLabId`/`boostCompound` (GH2O). The upgrader spawn request carries `memory.boosts = [{ part: WORK, compound: 'GH2O' }]` when the gate is open. `hauler.ts` `pickupBoostLab` has a flush guard: if the reserved lab holds a different mineral, the hauler withdraws it first before loading GH2O. `upgrader.ts` opts in with `if (!ensureBoosted(creep)) return;` at the top of `run()`.
- **`getColonyScore(room)`** (`src/utils/colonyPlanner.ts`) — heap-cached per-colony priority score, recomputed every 500 ticks (`SCORE_CACHE_INTERVAL`). Formula: `rclFactor × incomeRate × storageFactor` where `rclFactor = max(8 - rcl, 1)` (younger rooms score higher), `incomeRate = activeSources × SCORE_SOURCE_RATE (10)` (falls back to planned source count during bootstrap when no miners are alive), `storageFactor = clamp(stored / SCORE_INCOME_REFERENCE (20k), 0.1, 1.0)` (bottoms out at 0.1 so income-starved rooms still get a non-zero score). Higher score = worth investing upgrade time in first. Consumed by `upgradersNeeded` (young-colony investment gate), `sendEnergyToColonies` (receiver ranking), and `runVisuals` (header display). `getColonyScores()` returns all owned rooms sorted by score. Console: `colonies()` now prints scores alongside lifecycle state. Export `resetColonyScoreCache()` in tests' `beforeEach` to prevent cross-test contamination.
- **Hauler pickup priority** (`src/roles/hauler.ts`): committed target (continue previous task if still valid) → urgent responder (storage → spawn/ext when structures need energy, nearest hauler only; won't preempt a committed target within range 3) → large dropped energy (≥1000, decay-critical, preempts link drain) → lab flush/input/output (`pickupLabInput` checks storage first, falls back to terminal when storage is empty so labs don't deadlock when minerals route straight to terminal) → storage link (≥200 energy threshold — catches real link transfers, ignores scraps) → boost-lab service (tops up the reserved `boostLabId` with its `boostCompound` then energy, to `BOOST_LAB_MINERAL_TARGET`/`BOOST_LAB_ENERGY_TARGET`; active at RCL 7+ under the GH2O gate set by `reserveBoostLab`, with a flush guard that withdraws any wrong mineral first) → dropped energy (≥50) → dropped minerals (≥50, non-energy; skipped when room has no storage or terminal — young colonies with nowhere to deliver would get permanently stuck) → ruins/tombstones (energy preferred; non-energy skipped when room has no storage or terminal for the same reason) → **source-container leg** [**SHELVED feature — `Memory.haulerPool` is off by default and must stay off**; it conflicts with the task-commitment model (committed haulers ignore the assignment, unassigned extras re-converge via the legacy pick) and gave worse convergence in live validation — see todo.md Phase 6 / `haulerPool.ts` header. When the (dormant) flag is set, `assignHaulers(room)` in `src/managers/haulerPool.ts` pre-assigns each home hauler to a specific source container via greedy fill+proximity (fullest first, nearest hauler; 0-energy containers excluded; extras left to fall through), checked at both the "full source containers ≥1000" and "any source container" steps — if assigned and container has energy, withdraws from it; otherwise falls through to the legacy fullest-first selection. Flag off (the default) runs the legacy selection unchanged] full source containers (≥1000) → mineral container (non-energy) → any source container (sorted fullest-first) → factory batteries (`RESOURCE_BATTERY` in factory store when `factoryRecipe` set) → terminal minerals. Delivery: energy to factory when `factoryRecipe` active and storage > floor; batteries route to terminal (preferred) or storage. Non-energy minerals deliver to the reserved boost lab (when carrying its `boostCompound`), then to matching lab input if active reaction, then storage (below `MINERAL_STORAGE_FLOOR` 5k) or terminal; if neither exists (young colony), the mineral is dropped via `creep.drop()` as a last resort to prevent permanent DELIVER deadlock. Task commitment prevents flip-flopping: once a hauler picks a target, it finishes the trip before re-evaluating priorities. Controller container delivery requires ≥200 free capacity to prevent trivial top-off diversions. Mineral delivery fills storage to `MINERAL_STORAGE_FLOOR` (5k) before overflowing to terminal, so labs always have stock available without round-tripping through the market.

### Body scaling

`src/utils/body.ts` `buildBody(pattern, energy, maxRepeats?)` repeats a body pattern as many times as the room's `energyCapacityAvailable` allows. Spawner always passes `spawn.room.energyCapacityAvailable`, so creeps automatically grow as extensions get built — do not hardcode bodies. Specialized builders exist for miners (`buildMinerBody` — maximizes WORK, cap 6), upgraders (`buildUpgraderBody` — maximizes WORK, cap 15), and remote miners (`buildRemoteMinerBody` — WORK+MOVE pairs at 1:1 ratio for off-road travel, plus 1 CARRY for building containers, cap 5 WORK). Non-production roles are capped via `maxRepeats` to control spawning cost: haulers ×8 (1600 energy), remote haulers ×8 (1600 energy), harvesters ×4 (800), builders ×4 (1000), repairers ×4 (800).

### Defense

- `src/utils/threat.ts` scores hostile creeps by body parts (HEAL 250 > CLAIM 200 > RANGED_ATTACK 150 > ATTACK 80 > WORK 30; dead parts ignored). `pickPriorityTarget(room)` returns the top-scoring hostile using a composite score: `threatScore * 10,000 + towerEffectiveness * 1,000 - hits`. Tower effectiveness is range-weighted (1.0 at range ≤5, 0.25 at range ≥20, linear between) using average distance from all towers — this prevents wasting tower ammo on border creeps while closer hostiles roam free. Zero-threat hostiles are still targeted; closer ones are preferred.
- Focus-fire (`managers/towers.ts`) is the deliberate policy — closest-target fire lets healers keep attackers alive indefinitely.
- Safe mode activates only when a hostile with `threatScore > 0` is within range 5 of a spawn / storage / controller, so scouts don't burn a charge.
- `defendersNeeded(room)` is **tower-aware**: before counting defenders it checks whether the room's energised towers can solo the threat. If there are towers with energy ≥ `TOWER_ENERGY_COST` whose combined capacity (`towers × THREAT_PER_TOWER`, 500 each) covers the threat **and** the enemy can't out-heal their fire (hostile HEAL/tick `< towers × TOWER_DPS_ESTIMATE`, 300 each), it returns 0 — a lone invader a tower vaporises no longer spawns a wasted defender. Towers fire every tick regardless and safe mode remains the backstop, so this only removes redundant creeps. Beyond tower capacity, or against a healing squad, or with no towers, it falls back to `min(ceil(threat / THREAT_PER_DEFENDER 200), 4)`.
- `defenderComposition(room)` returns `{ melee, ranged, healer }` counts based on threat band: ≤200 → melee only; 201–600 → melee+ranged; >600 → melee+2ranged+healer. Bumps ranged by 1 when the enemy squad includes HEAL parts. Capped at 4 total defenders. Spawner queues `defender` (melee), `rangedDefender` (kites at range 3, mass-attacks clusters), and `healer` (follows nearest combatant, heals range 1 or rangedHeal range 3) accordingly.
- `src/utils/neighbors.ts` records every hostile player's name, attack count, max threat score, body signatures, and rooms visited into segment 5 (`NeighborRecord`). Classifies players as `passive` / `unknown` / `aggressive` (3+ attacks OR maxThreatScore ≥500). The remote planner rejects rooms where any player previously classified `aggressive` has been seen within 20k ticks (`evaluateRemoteRoom` calls `hostilesSeen(roomName, 20_000)` and filters by `getNeighbor(name)?.hostility === 'aggressive'`). Console: `neighbors()` prints the full table.
- **Hunter role** (`src/roles/hunter.ts`) — dispatches a melee creep to clear NPC Invaders from remote rooms and transit rooms (rooms on the path to a colony). Strictly targets creeps owned by `'Invader'`; never engages player creeps. State machine: `TRAVEL` paths to `(25,25)` of `targetRoom` and waits until `isInRoomInterior` is true (≥3 tiles from the border) before transitioning, preventing the work state starting on a border tile that the engine would auto-evict next tick; `HUNT` attacks the lowest-HP invader and self-heals each tick; on room clear it deletes `RoomMemory.invaderSeenAt` and transitions to `RETREAT`; `RETREAT` paths home and recycles at the spawn. Body uses `buildHunterBody` (3 tiers: <790 energy → not spawned; 790–1309 → `[TOUGH×2, MOVE×4, ATTACK×4, HEAL×1]`; ≥1310 → `[TOUGH×3, MOVE×6, ATTACK×6, HEAL×2]` — TOUGH absorbs hits, HEAL is last so it survives longest, road-speed in both tiers). Invader detection: `runDefense` sets `RoomMemory.invaderSeenAt = Game.time` for any visible room containing an `'Invader'` creep, and clears it when visibility confirms the room is clear. The spawner considers a room actively infested for `INVADER_MEMORY_TICKS` (500 ticks) after the last sighting. `huntersNeeded(room)` counts unique infested rooms (remote rooms + transit rooms of colonies) that lack a hunter. Hunters are spawned at Priority 1 (after defenders, before miners) so an invader camping a remote source doesn't block energy production indefinitely; one hunter per infested room, scoped by `homeRoom`.
- **Perimeter defense** (`src/utils/perimeterPlanner.ts`) — BFS flood-fill from all room exit tiles, stopping at a core zone of Chebyshev radius `CORE_RADIUS` (10) from the spawn anchor. All tiles reachable from exits without crossing the core are "exterior"; the perimeter = exterior tiles adjacent to interior/core tiles. Algorithm: seed BFS with passable border tiles → flood with 8-direction adjacency, blocking terrain walls and core-zone tiles → exterior tiles that border a non-exterior neighbour form the perimeter. ~80 LOC. `computePerimeter(room)` returns a `PerimeterPlanData` stored in `RoomMemory.perimeterPlan` (`version`, `coreRadius`, `perimeterTiles[]`, `gateTiles[]`, `gateTargets[]`). Invalidated automatically when `PERIMETER_PLAN_VERSION` changes or `remoteRooms` changes (ensures remote gate targets stay current). **Gate strategy**: sources outside the core, controller if distant, and one tile per remote room exit direction each get a 2-tile-wide gate (rampart only — passable by own creeps); all other perimeter tiles get walls (impassable, zero decay cost). `ensureRoomPlan()` in `roomPlanner.ts` calls `computePerimeter` lazily on first access or after invalidation.
- **Combat event logger** (`src/utils/combatLog.ts`) — ring buffer (max 100 entries) in `Memory.combatLog`. Events: `threat_appeared` (first hostile sighting), `threat_ended` (room clear), `safe_mode_activated`, `safe_mode_unavailable` (charged/on-cooldown status), `tower_energy_low` (any tower below 25% during combat). `logCombat(event)` appends + echoes to console. `formatCombatLog()` formats full log for the `combatLog()` console command. Useful for post-fight reviews when the room was unobserved.

### Profiling & visuals (opt-in)

Gated by Memory flags so production ticks pay ~nothing when off:

- `Memory.profiling = true` → `profile(name, fn)` records CPU deltas as exponential moving averages in `Memory.stats`.
- `Memory.visuals = true` → `runVisuals()` draws per-room RCL/energy/creep-count/CPU headers (including colony priority `score=X.X`), storage level, controller progress %, source-load markers, and idle creep indicators (grey circles).
- `Memory.profileOverlay = true` (requires `Memory.visuals`) → draws a sorted CPU stats table (top 12 entries from `Memory.stats` by average CPU, colour-coded: >3ms red, >1ms yellow, else green) on the first owned room. Gated separately so per-room headers can stay on without the extra table. Toggle from the console: `Memory.profileOverlay = true`.

When adding a new manager or hot path, wrap it in `profile('label', fn)` so it surfaces in `stats()`.

Console-callable exports from `main.ts`: `stats()`, `resetStats()`, `status()`, `replanLayout(roomName)`, `replanPerimeter(roomName)`, `combatLog()`, `neighbors()`, `suggestSpawn(roomName)`. The Screeps console evaluates against `global` in IVM — to expose a new console command, add an `export const` in `main.ts` and register it on `global`. `suggestSpawn(roomName)` scores all open tiles in a room using terrain only (no live structures) and returns the best spawn position for claim planning, storing the result in `RoomMemory.suggestedSpawnPos`. `replanPerimeter(roomName)` clears `RoomMemory.perimeterPlan` and recomputes it immediately, returning a summary string (tile counts, gate count, gate targets). `combatLog()` formats and prints the full combat event ring buffer for post-fight review.

### Creep state machine

All roles use `src/utils/stateMachine.ts`. Each role defines a `StateMachineDefinition` — a `Record<string, StateHandler>` where each handler's `run(creep)` returns a state name to transition or `undefined` to stay. State is persisted in `creep.memory.state`. The engine validates the state exists (falls back to default on code deploy with renamed states) and calls optional `onEnter()` on transitions. Inspect `creep.memory.state` in-game to see what any creep is doing.

### Movement & traffic

`src/utils/movement.ts` provides a `moveTo` wrapper used by all roles. Each creep computes its own path via `PathFinder.search` and issues `creep.move(direction)` directly — there is no centralised intent queue. Traffic avoidance is handled entirely through the CostMatrix (`src/utils/trafficManager.ts`, cached per room per tick): roads cost 1, moving friendly creeps cost 0 (not added to the default matrix — their soft-avoid penalty inflated corridors near idle hauler clusters and pushed PathFinder onto longer detours), stationary creeps (miners) cost 255 (hard block — paths must go around), impassable structures cost 255, hostile creeps cost 255. Stationary roles call `registerStationary(creep, PRIORITY_STATIC)` to mark their tile as impassable in the CostMatrix. The stuck-detection repath callback (`pathRoomCallbackAvoidCreeps`) still applies cost 50 to moving creeps so the recovery path actually routes around active blockers. The Screeps engine handles 2-way swaps natively when both creeps move into each other's tile. Always use `moveTo()` instead of direct `creep.moveTo()` or `creep.move()`.

**Active blocker pushing**: after resolving the next path step, `pushBlocker` in `trafficManager.ts` calls `room.lookForAt(LOOK_CREEPS, nextPos)`. If a friendly, non-stationary creep is there and its `movePriority` (from `CreepMemory.movePriority`, defaulting to `PRIORITY_DEFAULT = 10`) is ≤ the mover's priority, it is nudged to any free adjacent tile (cost < 255) via a single `move()` call. Creeps are only pushed once per tick (tracked in `pushedThisTick`, cleared by `resetTraffic()`). This is a best-effort nudge — skipped silently when no free neighbour exists.

**Stuck detection**: after 2 ticks at the same position `moveTo` calls `executeMoveAvoidCreeps` (cost-50 repath). After 3 ticks it forces another `executeMoveAvoidCreeps` with a fresh cache invalidation and resets the counter, then retries every 3 ticks until the creep moves. The old native `creep.moveTo` fallback has been removed — all movement stays inside the traffic system.

For cross-room movement, `getPath` uses `maxRooms: 16` and `maxOps: 10000` when the target is in a different room — enough headroom for diagonal / depth-3 hops (e.g. a scout heading to a SE-diagonal room). In-room searches use `maxOps: 5000` (raised from 2000) to avoid partial paths in cluttered rooms. The `roomCallback` (`pathRoomCallback` in `trafficManager.ts`) returns per-room CostMatrix when visibility exists, returns `false` (skip) for unseen rooms whose `scoutedOwner` is another player so we don't path creeps into enemy tower range, and returns an empty CostMatrix for all other unseen rooms so PathFinder falls back to terrain costs. Our own `scoutedOwner` is compared against `Game.spawns[*].owner.username` so a brief vision loss into one of our rooms doesn't make it impassable. Remote roles path directly to stored source positions via `RoomPosition` constructed from Memory coordinates — no `Game.map.findExit` needed.

When a role's `moveTo` target requires standing on a specific tile (e.g. miner on container), pass `range: 0` — the default range is 1, and the traffic manager skips movement when already in range.

`isInRoomInterior(creep)` (also exported from `movement.ts`) returns `true` when the creep is at least 3 tiles from every border edge (`x > 2 && x < 47 && y > 2 && y < 47`). Use this in TRAVEL states to avoid starting work logic on border tiles: a creep ending a tick on a border tile can be auto-evicted to the adjacent room by the Screeps engine. The hunter's `TRAVEL → HUNT` transition uses this check.

### Remote mining

Remote mining sends creeps to harvest sources in adjacent unowned rooms. Gated by miner economy (containers must be built first). The flow:

1. A `scout` (1 MOVE) explores adjacent rooms and records source count, ownership, hostile presence, and source positions (`scoutedSourceData`) in `RoomMemory`. Re-scouts rooms every 5000 ticks. Only spawned when `findScoutTarget()` finds an unscouted or stale room **and** the colony is below its storage-gated remote cap (`remoteRoomsBelowCap`: `remoteRooms.length < remoteRoomCap(room)`) — a colony already at its cap has no use for more remotes, so it stops scouting rather than burning spawn bandwidth re-scouting depth-3 territory it cannot exploit. Scouting auto-resumes if a remote is lost or storage grows past the next scale threshold. No permanent scout is kept alive.
2. `selectRemoteRooms()` (`src/utils/remotePlanner.ts`) evaluates scouted rooms every 100 ticks — rejects owned rooms, rooms reserved by other players (own reservations are accepted), sourceless rooms, rooms with recent hostile sightings (< 1500 ticks old), and rooms where aggressive neighbors have been observed within 20k ticks. Picks best rooms by source count up to a storage-gated cap with hysteresis: 1 remote below `REMOTE_ROOM_SCALE_THRESHOLD` (100k storage energy), 2 once storage clears that bar; the second remote is kept until storage drops below 70k (`SCALE_DOWN_THRESHOLD = REMOTE_ROOM_SCALE_THRESHOLD * 0.7`), preventing churn when storage oscillates near the threshold. Sets `RoomMemory.remoteType` (`'reserved'` for rooms with a controller, `'remote'` for highway/sourceless-controller rooms, `'keeperRoom'` for Source Keeper rooms) and initialises `RoomMemory.defensePolicy` (`'defend'` for reserved, `'flee'` for remote/keeperRoom) if not already set. **SK room opt-in**: passes `allowKeeperRooms: energyCapacityAvailable >= 5300` to `evaluateRemoteRoom`; SK rooms score `sourceCount * 3` when a keeper killer is assigned and alive, otherwise remain rejected. Source Keeper creeps (`owner.username === 'Source Keeper'`) are excluded from the flee scan in `remoteThreat.ts` for keeper rooms — the killer handles them; player hostiles still trigger flee.
3. `ensureRemoteRoomPlan()` scans sources in remote rooms we have visibility into; bootstraps source data from `scoutedSourceData` when no visibility.
4. Remote miners (reusing `miner` role with `targetRoom` set) path directly to stored source positions via cross-room PathFinder. They harvest at the source and build their own container (they have 1 CARRY part for this). Once the container is built, the miner also repairs it when damaged (using the same CARRY buffer — home miners use theirs for link transfers instead). Before the container is built, energy drops on the ground for haulers. Remote miners are pre-spawned 150 ticks before their predecessor's TTL expires (`REMOTE_MINER_PRESPAWN_TICKS`) so cross-room travel doesn't leave the source unmined.
5. `remoteHauler` creeps pick up dropped energy or withdraw from containers in the remote room, then deliver to storage/spawns/towers/controller container in the home room. When idle in the remote room, they wait near the source to avoid border-tile bouncing. Opportunistic loot pickup (ruins/tombstones) is **energy-only** — the DELIVER state only transfers `RESOURCE_ENERGY`, so picking up non-energy minerals would leave them permanently trapped in the hauler's store for its lifetime.

`CreepMemory.homeRoom` identifies which room a remote creep belongs to. `CreepMemory.targetRoom` tells it which room to operate in. Local miners have neither field set.

6. A `reserver` (`[CLAIM×2, MOVE×2]`, 1300 energy) continuously calls `reserveController()` on a remote room's controller. 2 CLAIM parts add +2 reservation ticks per call (net +1/tick after decay), doubling source capacity (1500 → 3000 energy/regen) and halving container decay. Spawned 1 per remote room where `remoteType === 'reserved'`. Scout records controller presence during room scanning.
7. `remoteBuilder` (`[WORK, CARRY, MOVE, MOVE] × 4`, 1000 energy) travels to the remote room, gathers energy from containers/drops, and builds road construction sites placed by `placeRemoteRoads`. Falls back to repairing damaged roads (< 75% HP). Spawned 1 per remote room when construction sites exist OR roads are below 50% HP. Temporary role — stops spawning once roads are built and healthy.

Future concerns for claiming are tracked in `todo.md`.

### Idle creep management

`src/utils/idle.ts` provides `markIdle(creep)` for roles that have no work to do. It registers the creep as idle (for visual indicators), parks idle creeps away from the spawn cluster, and recycles chronically idle creeps. Roles that can go idle: hauler (nothing to pick up or deliver), harvester (all delivery targets full), defender/rangedDefender/healer (no hostiles). Builders, repairers, and upgraders always fall back to upgrading the controller so they never idle. When `Memory.visuals` is enabled, idle creeps are marked with a grey circle overlay.

**Parking**: if the room's controller is ≥8 tiles from the spawn, idle creeps rally to the controller position (far from the extension/lab/spawn cluster). Each creep gets a deterministic tile offset based on a hash of its name (`SPREAD_OFFSETS`, 2-tile spacing) so they don't pile up. Falls back to range 5 of storage or spawn (was range 3) when the controller is too close.

**Recycle**: `idleSince` + `_idleLastTick` in `CreepMemory` track consecutive idle ticks (streak resets on any gap where the creep wasn't idle). Thresholds: defender/rangedDefender/healer 100 ticks → recycled only if no threat seen within 200 ticks (keeps combat roles alive during active fights). Recovers ~50% of body cost as spawn energy when recycled. Haulers do **not** recycle — the old 50-tick hauler threshold churned haulers between spawn/recycle cycles during normal idle gaps, costing more energy than it saved; haulers now stay alive through brief idle periods and resume work when pickups appear.

### Deployment

`scripts/deploy.mjs` POSTs `dist/main.js` to `https://screeps.com/api/user/code` using `X-Token` auth. Config lives in `.env` (gitignored); copy `.env.example` to get started. After a successful deploy, the script lists all branches and warns if the target branch isn't the active world branch.

The rollup build stamps a version banner (`// screepsAI v{version} - built {timestamp}`) on line 1 of `dist/main.js`, read from `package.json`. `npm run deploy` auto-bumps the patch version before building.

### Error mapping

`src/utils/ErrorMapper.ts` wraps the main loop with `wrapLoop`. It uses a custom synchronous VLQ decoder (not the `source-map` package, which is async and too slow for Screeps) to map runtime errors back to TypeScript source lines. The bundled `main.js.map` is loaded via Screeps' `require('main.js.map')`; the parsed map is cached across ticks and rebuilt on global reset.

## Testing

Tests live in `test/` mirroring the `src/` structure. Vitest is the runner, configured in `vitest.config.ts` with `globals: true` (no imports needed for `describe`/`it`/`expect`).

`test/mocks/screeps.ts` is a setup file that injects Screeps constants (`WORK`, `FIND_STRUCTURES`, etc.) and provides `mockCreep()`, `mockRoom()`, and `resetGameGlobals()` factory helpers. Call `resetGameGlobals()` in `beforeEach` when tests mutate `Game` or `Memory`.

**When to write tests:** When adding or modifying utility functions, manager logic, or role state machines, add or update corresponding tests. Pure logic (no Screeps runtime dependency) is highest priority. Functions that need only light mocking (mock creep/room) are also good candidates. Skip tests for code tightly coupled to the Screeps runtime (construction placement, error mapping, the main loop). Well-tested modules: `sources.ts`, `remotePlanner.ts`, `roomPlanner.ts`, `scout.ts`, `miner.ts`, `threat.ts`, `body.ts`, `stateMachine.ts`, `spawner.ts`, `labs.ts`, `hauler.ts`, `remoteHauler.ts`, `construction.ts`, `terminal.ts`, `reserver.ts`, `neighbors.ts`, `layoutPlanner.ts`, `factory.ts`, `remoteThreat.ts`, `throttle.ts`.

`resetGameGlobals()` now also stubs `RawMemory` (with a `segments` record and `setActiveSegments` spy), so segment-backed utilities like `neighbors.ts` can be tested. Tests that exercise segments across multiple fake ticks should call `flushSegments()` between ticks to flush the dirty-flag write-back (the game loop does this at the end of each tick).

**To make internal functions testable:** Export them. The spawner's `*Needed()` functions and `buildSpawnQueue()` are exported specifically for testing.

## TypeScript / Screeps specifics

- `tsconfig.json` uses `strict` + `noUncheckedIndexedAccess`. `Game.creeps[name]` and similar index accesses return `T | undefined` — always null-check.
- `lib` is `ES2021` (no DOM). A minimal `console` and `require` are declared in `src/types.d.ts` for the Screeps sandbox globals. Do not add `@types/node`.
- The rollup bundle marks `lodash` as external because Screeps provides it globally. Do not import lodash in new code — use native `Object.values` / array methods (this was a deliberate cleanup).
- `"type": "commonjs"` in `package.json`; rollup outputs CJS because the Screeps VM is CJS.
- Screeps IVM uses `global` (not `globalThis`). `global` is declared in `src/types.d.ts`. `module` is also declared there for the console export pattern.
- Do not add `@types/node` — the Screeps VM is not Node. Minimal globals (`console`, `require`, `global`, `module`) are declared in `src/types.d.ts`.

## Engineering lessons (hard-won)

These were distilled from real mistakes — hauler pool, traffic regressions, layout bugs. Apply them before shipping anything non-trivial.

1. **External insights don't always translate.** Approaches that work well in other bots or systems may conflict with this codebase's architecture. Understand *why* something works elsewhere before porting it — context matters more than the idea itself.

2. **Live validation is the ground truth.** Unit tests pass, logic looks correct, and it still breaks in production. The Screeps environment (tick timing, partial visibility, emergent creep interaction) surfaces edge cases no mock captures. Validate non-trivial changes live before calling them done. This has saved us at least three times.

3. **Feature flags are worth the boilerplate.** Hiding risky or experimental work behind a `Memory.*` toggle costs almost nothing and buys everything: the feature can ship dormant, be tested without affecting production, and be shelved without reverting. Shelved code must carry a comment explaining *why* it's off and what it conflicts with — see the hauler pool note in `hauler.ts` and CLAUDE.md as the model.

4. **Respect existing architectural invariants.** Before replacing a system, understand what it is already solving. The hauler pool conflicted with task-commitment (which was already doing implicit per-hauler assignment). The new approach didn't fail because the idea was wrong — it failed because it didn't account for an invariant the existing code depended on.

5. **Measure before you optimise.** "Convergence looks bad visually" is intuition, not evidence. If there's no proxy metric (e.g. ticks a source container stays full, average hauler idle rate), you cannot know whether a change is winning. Define the metric first, then change the system.

6. **Complexity has a carrying cost.** Every file added must be read, understood, and kept in sync by future work — including dormant code. A simpler system that is slightly suboptimal usually beats a complex system that is theoretically better. Prefer the simplest thing that solves the actual problem at the current scale.

7. **Emergent solutions can be more robust than designed ones.** Task-commitment wasn't designed as a dispatch system — it emerged from fixing flip-flopping. Designed systems (greedy-fill-by-proximity) have sharp edges in dynamic tick-by-tick environments that an emergent pattern may already have smoothed over through iteration. Before replacing emergent behaviour, be sure you understand all of what it is doing.

8. **Annotate shelved code thoroughly.** When something is shelved rather than deleted, the comment is the only thing preventing future-you from re-implementing the same mistake. Include: what it was trying to do, why it was shelved, what it conflicts with, and where to look for context. The hauler pool notes in CLAUDE.md and `hauler.ts` are the reference model for this.
